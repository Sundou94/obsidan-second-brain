## 1. 과제 목표

- 테스트 데이터셋 기준으로 `F1-Score 0.70 이상`을 달성한다.


## 2. 베이스라인 문제점
### 2-1. 지표 불균형 원
베이스라인 모델의 저장 결과는 아래와 같았다.

```
베이스라인 모델이란? 

베이스라인 모델은 데이터 분석 및 머신러닝 프로젝트에서 초기 비교 기준을 제공하는 간단한 모델입니다. 이 모델은 주로 복잡한 알고리즘을 사용하기 전에 기본 성능 수준을 설정하는 데 사용됩니다. 이를 통해 나중에 개발된 복잡한 모델의 성능이 어느 정도 개선되었는지를 평가할 수 있습니다.

베이스라인 모델은 일반적으로 다음과 같은 특성을 가집니다:

1. **단순함**: 베이스라인 모델은 가능한 한 간단하게 설계됩니다. 예를 들어, 회귀 문제의 경우 평균값을 예측하는 모델이나, 분류 문제의 경우 가장 빈번한 클래스를 예측하는 모델이 될 수 있습니다.

2. **빠른 구현**: 베이스라인 모델은 빠르게 구현 및 평가할 수 있어야 합니다. 이는 프로젝트 초기 단계에서 효율적인 비교 기준을 제공하기 위함입니다.

3. **비교 기준 제공**: 베이스라인은 더 복잡한 모델의 성능을 비교할 때 기준점 역할을 합니다. 새로운 알고리즘이나 기법들이 베이스라인보다 성능이 좋지 않다면, 해당 방법들은 실질적인 가치가 없다고 판단될 수 있습니다.

4. **결정 요인**: 베이스라인에서 좋은 결과를 얻지 못하면 데이터 전처리나 문제 정의에 문제가 있을 가능성이 있으므로, 이를 통해 데이터 및 문제 설정 자체를 재검토할 필요가 있을 수 있습니다.

AI4I 2020 Predictive Maintenance 과제에서는 이러한 베이스라인 모델을 설정하여, 장비 고장의 조기 예측 능력을 향상시키기 위한 기본 성능 수준을 측정할 수 있을 것입니다. 이를 통해 더 발전된 예측 유지보수 시스템 개발 시 초기 성능 비교가 가능해집니다.
```





| 모델                   | Threshold | Accuracy | Precision | Recall |     F1 | ROC-AUC |     |
| -------------------- | --------- | -------: | --------: | -----: | -----: | ------: | --- |
| Baseline saved model | 0.50      |   0.9205 |    0.2903 | 0.9265 | 0.4421 |  0.9773 |     |
정확도(Accuracy) : 전체 데이터 중에서 AI 모델이 올바르게 예측한(정답을 맞춘) 데이터의 비율
정밀도(Precision) : AI 모델이 '양성(Positive)'이라고 예측한 데이터 중에서 실제로 양성인 데이터의 비율
재현율(Recall) : 실제 '양성(Positive)'인 전체 데이터 중에서 AI 모델이 놓치지 않고 양성이라고 찾아낸 데이터의 비율
```
Threshold 란 무엇인가? 

Predictive Maintenance 과제에서 "Threshold"는 일반적으로 예측 모델이 작동하는 기준점을 의미합니다. 구체적으로, 장비나 시스템의 상태를 모니터링하여 고장 가능성을 예측할 때, "Threshold"는 경고나 조치를 취하기 위한 임계값을 설정하는 데 사용됩니다. 

예를 들어, 특정 센서 데이터(온도, 진동 등)가 이 임계값을 초과하면 장비 고장을 예측하고 예방적 유지보수를 수행해야 하는 시점으로 간주할 수 있습니다. 이러한 Threshold는 통계적 분석이나 과거 데이터에 기반한 머신러닝 모델을 통해 결정되며, 적절한 Threshold 설정은 유지보수의 효율성과 비용 절감에 크게 기여할 수 있습니다.

AI4I 2020 Predictive Maintenance 프로젝트에서는 이러한 Threshold를 적절히 설정하여 장비의 고장을 미리 감지하고 예방 조치를 취함으로써 시스템의 가동 시간을 최대화하고 유지보수 비용을 최소화하는 것이 목표일 것입니다.
```

임계값 이구나..?

정확도와 ROC-AUC는 높지만, `Precision 0.2903`이 매우 낮아 `F1-Score`가 크게 떨어진다.  
즉, ==실제 고장은 대부분 잡았지만 정상 샘플까지 고장으로 과다 판정한 상태다.==



### 2-2. 원인 해석

1. `WeightedRandomSampler`로 학습 배치를 거의 50:50에 가깝게 만들면서 실제 분포와 학습 분포가 달라졌다.
2. 테스트셋 실제 분포는 정상 96.61%, 고장 3.39%인데, 베이스라인은 `0.5` 고정 threshold를 사용했다.
3. 검증셋 없이 테스트셋과 다른 판단 기준으로 학습해 `False Positive`가 과도하게 증가했다.

  

결과적으로 베이스라인은 `Recall 중심` 모델이 되었고, 설비 운영 관점에서는 오경보가 많아 실사용성이 떨어진다.

  

## 3. 성능 개선 방법론

  

### 3-1. 데이터 분할 구조 개선

- 기존: Train / Test

- 개선: Train / Validation / Test = 60 / 20 / 20

  

적용 이유:

- threshold를 테스트셋이 아니라 검증셋에서 선택해 데이터 누수를 막기 위함

- 조기 종료와 모델 선택 기준을 검증셋에 두기 위함

  

### 3-2. 불균형 처리 방식 변경

- 제거: `WeightedRandomSampler`

- 적용: `BCEWithLogitsLoss(pos_weight=28.5567)`

  

적용 이유:

- 실제 데이터 분포는 유지하면서 손실 함수에서 소수 클래스의 중요도만 높이는 방식이 더 안정적이다.

- 배치 분포를 인위적으로 50:50으로 바꾸지 않아 Precision 붕괴를 줄일 수 있다.

  

### 3-3. 도메인 기반 파생 특성 추가

추가한 특성:

- `Temp difference [K]`

- `Mechanical power [W]`

- `Torque wear interaction`

- `Wear speed ratio`

- `Process/Air temp ratio`

  

적용 이유:

- AI4I 데이터의 고장 메커니즘은 단일 센서값보다 `온도 차`, `동력`, `토크-마모 상호작용`처럼 관계형 특성에서 더 잘 드러난다.

- MLP가 원시 변수만 볼 때보다 고장 패턴을 더 쉽게 분리할 수 있다.

  

### 3-4. MLP 구조 및 학습 방식 개선

- 구조: `64 -> 32 -> 1`

- 적용: `BatchNorm`, `Dropout(0.1)`, gradient clipping, early stopping

- 학습률: `0.002`

  

적용 이유:

- 베이스라인보다 표현력은 키우되 과적합은 억제하기 위함

- 검증 F1 기준 조기 종료로 최고 성능 시점의 모델만 채택하기 위함

  

### 3-5. Threshold 최적화

- 베이스라인: `0.50` 고정

- 개선: 검증셋에서 F1가 최대가 되는 threshold 탐색

- 최종 선택 threshold: `0.95`

  

적용 이유:

- 불균형 데이터에서는 `0.5`가 최적 threshold가 아닐 가능성이 크다.

- Precision-Recall trade-off를 조정해 F1를 직접 개선할 수 있다.

  

## 4. 개선 결과

  

| 모델 | Threshold | Accuracy | Precision | Recall | F1 | ROC-AUC |

|---|---:|---:|---:|---:|---:|---:|

| Baseline saved model | 0.50 | 0.9205 | 0.2903 | 0.9265 | 0.4421 | 0.9773 |

| Improved model, threshold 0.50 | 0.50 | 0.9240 | 0.2900 | 0.8529 | 0.4328 | 0.9724 |

| Improved model, tuned threshold | 0.95 | 0.9825 | 0.7705 | 0.6912 | 0.7287 | 0.9724 |

  

핵심 해석:

- threshold를 그대로 `0.5`로 두면 개선 모델도 F1가 낮다.

- 즉, 이번 과제의 핵심 병목은 `모델 자체`만이 아니라 `판단 임계값`에도 있었다.

- 최종적으로 Precision이 크게 개선되면서 테스트셋 기준 `F1 = 0.7287`을 달성했다.

  

## 5. 결론

- 베이스라인의 취약점은 `불균형 데이터에서 Recall 편향이 심하고 Precision이 붕괴`된다는 점이다.

- 개선의 핵심은 `실제 분포 유지 + 손실 가중치 + 파생 특성 + 검증셋 기반 threshold 최적화`였다.

- 최종 모델은 테스트셋 기준 목표였던 `F1-Score 0.70 이상`을 만족했다.
