### I. 데이터셋 구축

#### AI 학습 데이터와 데이터셋
AI 학습 데이터란 인공지능이 문제를 해결하고 결과를 내기 위해 사용하는 **모든 종류의 데이터**를 의미한다. 원재료(일반 데이터)를 AI 학습 재료로 만들기 위해 **1차 필터링 정제(불필요한 오류 제거)** 및 **2차 라벨링(가공)** 과정을 거친다.

#### AI 학습 방식의 4가지 분류
| 학습 방식 | 데이터 형태 | 특징 |
|---|---|---|
| 지도학습 (Supervised Learning) | 문제(입력)-정답(출력) 쌍 | 입력에 대한 정답이 있는 데이터로 학습 |
| 비지도학습 (Unsupervised Learning) | 입력만 존재 | 정답 없이 데이터 자체의 패턴을 스스로 탐색 |
| 준지도학습 (Semi-supervised Learning) | 일부는 입력-정답 쌍, 나머지는 입력만 | 일부 라벨 데이터 + 대량의 비라벨 데이터 활용 |
| 강화학습 (Reinforcement Learning) | 상태-행동-보상 | 행동에 대한 보상을 통해 AI가 더 나은 행동 학습 |

#### LLM 학습 유형(활용 목적별)
- **사전 학습(Pre-training)**: 정답이 없는 대규모 비지도 학습 데이터로 기초 지식 습득
- **미세조정(Fine-Tuning)**: 특정 Task에 맞는 입력-출력 쌍 데이터로 능력 강화
- **인스트럭션 튜닝(Instruction Tuning)**: 사용자 명령어에 지시를 따르도록 학습
- **RLHF(인간 피드백 기반 강화학습)**: 사람이 더 나은 답변을 선택하도록 훈련
- **안전성 튜닝(Safety Tuning)**: 위험·편향·비윤리적 문장 필터링

#### 학습 데이터셋 구축 절차
구축 순서: **구축계획 수립 → 데이터 획득·수집 → 데이터 정제 → 데이터 가공 → 데이터 학습**

- **데이터 획득·수집**: 크라우드소싱 또는 전문 인력을 통해 직접 생산하거나 기존 시스템에서 확보. 특정 범주에 편중되지 않도록 균형 있게 수집하고, 정제·가공 손실을 고려해 실제 필요량보다 충분히 많이 확보
- **데이터 정제**: 원시데이터를 AI 학습에 필요한 형식·크기로 조정하고, 중복 제거·비식별화를 거쳐 **원천데이터**를 확보하는 활동. 정제된 원천데이터는 생성형 AI에서 **자기 지도 학습(Self Supervised Learning)**으로 활용 가능
- **데이터 가공**: 원천데이터에 AI 학습 목적에 맞는 **Ground Truth(정답)**과 주석을 부여하여 학습데이터로 전환. 데이터셋은 과적합 방지를 위해 학습용(80%)/검증용(10%)/시험용(10%) 비율로 분할
- **데이터 학습**: 사전학습 후 데이터 증강·최적화·미세조정을 통해 성능 향상. 모델 설계 → 성능 평가 → 품질검증 및 보완 순으로 진행

#### 데이터 유형별 구축 절차
- **텍스트 데이터**: 수집 방식(API, 웹 크롤링, 크라우드소싱) → 품질 확보 전략(불필요한 철자 제거, 틀린 맞춤법 정제, 문장 부호 교정, AI 생성 문장 검토) → 가공(NER, 기계번역, 감성분석, 질의응답, 텍스트 분류 등)
- **이미지 데이터**: 이미지 정제 기준(해상도, 중복, 노이즈, 편향, 포맷, 개인정보, 저작권) → 가공 방식(바운딩 박스, 키포인트, 폴리라인, 폴리곤, 시맨틱/인스턴스 세그멘테이션, 태깅)
- **영상 데이터**: 정제 기준(해상도, 프레임 손상, 소음·잡음, 영상 길이, 불필요한 장면) → 가공(바운딩 박스, 키포인트, 폴리라인, 폴리곤, 시퀀스 세그멘테이션)
- **오디오 데이터**: 정제 기준(음량, 발음, 소음·잡음, 잘림, 안들림) → 가공(전사: 일반/이중/화자 전사, STT, 화자 인식·분할) → 오디오 학습 모델(Wav2Vec, Whisper)

### II. 제조 데이터 표준

#### 표준의 정의
- **사전적 의미**: 사물의 정도를 정하는 목표나 기준
- **국제적 통념(ISO/IEC Guide 2:2004)**: 최적의 상태 도달을 목적으로 공인된 기구에 의해 합의에 기초하여 수립된 것
- **국가적 통념**: 이익·편리가 공정하게 얻어지도록 통일·단순화를 꾀한 결정(한국산업표준, 국가기술표준원)

#### 표준화 기구의 구분
- **공식 표준화기구(De-Jure)**: ISO, IEC, ITU 등 국제적으로 공인된 기구. 광범위한 기술 분야 표준화
- **사실 표준화기구(De-Facto)**: 기업간 연합체. 유사 기술 분야에서 다수 연합체가 표준화 경쟁

#### RAMI 4.0 참조 모델
독일 플랫폼 인더스트리 4.0(Plattform Industrie 4.0) 연합과 ZVEI, VDI/VDE가 개발한 **Industry 4.0을 위한 참조 아키텍처 모델(Reference Architectural Model Industrie 4.0)**. 독일 국가 표준 **DIN SPEC 91345**로 제정되었으며, 이후 **IEC/PAS 63088** 글로벌 규격으로 채택.

**3축 구조:**
- **X축 (제품 수명주기 및 가치 흐름)**: 자산의 시간적 생애주기와 가치 창출 흐름 → IEC 62890. **AAS(Asset Administration Shell)**가 핵심 표준으로, Type(설계) → Instance(운영) 전 수명주기 디지털 트윈 추적
- **Y축 (계층 구조/Hierarchy Levels)**: 전통적인 자동화 피라미드(ISA-95)를 스마트 팩토리로 확장한 수직적 통합 → IEC 62264(IT/제어 통합), IEC 61512(배치 제어 및 공정 관리). **OPC-UA**가 하위 계층(센서·PLC)에서 상위 계층(엔터프라이즈·클라우드)으로 수직 통합하는 핵심 통신 표준
- **Z축 (아키텍처 계층/Architecture Layers)**: 물리적 하드웨어 자산이 디지털 IT 세계로 연결·추상화되는 6계층 구조 → Asset / Integration / Communication / Information / Functional / Business

#### 인더스트리 3.0 vs 4.0
- **인더스트리 3.0**: 특정 물리적 장비에 종속된 기능, 고립된 수직적 통신 체계 (Data Silo)
- **인더스트리 4.0**: CPS(사이버 물리 시스템) 기반으로 이기종 시스템 간 원활한 데이터 교환. 모든 이해관계자가 동일한 용어로 소통할 수 있는 거시적인 '표준 지도(Map)' 필요

### III. OPC-UA

#### OPC-UA 정의
**Open Platform Communications Unified Architecture**. 공장 내 설비(로봇, PLC, 센서 등) 간 호환성 부족과 데이터 전달 어려움을 해결하기 위해, 산업 자동화 분야에서 **안전하고 신뢰성 있게 데이터를 교환하기 위해 만들어진 국제 표준 통신 프로토콜**.

#### OPC-UA 아키텍처의 2계층
- **Foundation Layer**: 시스템 간 연결 방법과 데이터 교환 규칙 정의 (Transport: Webservice/OPC UA Binary, OPC UA Data Model: Modeling Rules, Base Services)
- **Information Models**: 탄탄한 인프라 위에서 실제 공장 운영에 필요한 4가지 핵심 기능 구현 (DA, AC, HA, Prog)

#### OPC-UA 통신 방식 비교
| 구분 | 클라이언트-서버 방식 | PubSub 방식 |
|---|---|---|
| 통신 형태 | 1:1 (Request-Response) | 1:N / N:N (Broadcast/Multicast) |
| 연결 상태 | 연결 지향형 (연결 유지) | 비연결 지향형 (연결 상태 유지 X) |
| 주요 사용처 | HMI/SCADA - PLC 제어, 정밀 장비 제어 | 공장 내 수천 개 센서 데이터 수집, 장비 간 고속 동기화 |

#### OPC-UA 4대 핵심 규격 (DA, HA, AC, Prog)
- **DA (Data Access)**: 현재의 온도, 압력 등 **실시간 센서 값**을 읽고 쓰는 기본 기능
- **HA (Historical Access)**: 서버 메모리나 DB에 저장된 **과거 데이터 포인트**를 기간 조건으로 조회. 트렌드 분석 활용
- **AC (Alarms & Conditions)**: 설비의 특정 조건(예: 온도 80도 초과) 충족 시 클라이언트로 **즉각적인 이벤트를 비동기적으로 push**
- **Prog (Programs)**: 클라이언트가 서버(설비)에 정의된 특정 함수를 실행하도록 **원격으로 명령** (예: 설비 리셋, 긴급 정지)

#### 데이터 품질 5대 지표
- **유일성**: 타임스탬프 기준 중복 레코드 비율 계산
- **완전성**: 온도 및 압력 컬럼의 최대 결측치 비율 확인
- **유효성**: 물리적 정상 범위(온도 0~100, 압력 0~5)를 이탈한 데이터 비율 평가
- **일관성**: 라벨링 조건식과 실제 부여된 라벨 간의 논리적 모순이 있는지 검증
- **정확성**: 타임스탬프 및 수치형 데이터 타입의 포맷 적합성 확인

## 키워드 정리

| # | 키워드 | 핵심 정의 |
|---|--------|-----------|
| 1 | AI 학습 데이터 | AI가 문제를 해결하고 결과를 내기 위해 사용하는 모든 종류의 데이터 |
| 2 | 지도학습 (Supervised Learning) | 입력-정답 쌍 데이터로 학습하는 방식 |
| 3 | 비지도학습 (Unsupervised Learning) | 정답 없이 입력 데이터의 패턴을 스스로 탐색하는 학습 방식 |
| 4 | 준지도학습 (Semi-supervised Learning) | 일부 라벨 데이터와 다수의 비라벨 데이터를 함께 사용하는 학습 방식 |
| 5 | 강화학습 (Reinforcement Learning) | 상태-행동-보상 구조로 더 나은 행동을 학습하는 방식 |
| 6 | Pre-training (사전 학습) | 비지도 대규모 데이터로 AI의 기초 지식을 습득시키는 첫 번째 학습 단계 |
| 7 | Fine-Tuning (미세조정) | 사전 학습된 모델을 특정 Task에 맞는 입력-출력 쌍으로 특화시키는 단계 |
| 8 | RLHF | Reinforcement Learning from Human Feedback. 사람의 선호를 반영하여 모델을 개선하는 강화학습 기법 |
| 9 | 원시데이터 | 수집 단계에서 획득한, 아직 정제되지 않은 날 것의 데이터 |
| 10 | 원천데이터 | 정제 단계를 거쳐 확보된, 라벨링 가공 이전의 데이터 |
| 11 | Ground Truth | 데이터 가공 단계에서 학습 목적에 맞게 부여되는 정답(레이블) |
| 12 | Self Supervised Learning | 정제된 원천데이터를 라벨 없이 사전학습/미세조정에 활용하는 자기 지도 학습 |
| 13 | 표준 (Standard) | 특정 목적을 위해 제품·서비스·시스템 등의 기술적 요구사항을 규정한 규칙·가이드라인·사양 |
| 14 | De-Jure 표준 | ISO, IEC 등 국제적으로 공인된 기구에 의해 수립된 공식 표준 |
| 15 | De-Facto 표준 | 기업 연합체가 시장에서 사실상 표준으로 채택된 비공식 표준 |
| 16 | RAMI 4.0 | Industry 4.0을 위한 참조 아키텍처 모델. DIN SPEC 91345 → IEC/PAS 63088으로 채택 |
| 17 | AAS (Asset Administration Shell) | 자산 관리 셸. RAMI 4.0 X축 핵심 표준으로, 설비의 전 수명주기 디지털 트윈 정보를 담는 표준 |
| 18 | ISA-95 | 전통적인 자동화 피라미드(IO → PLC → SCADA → MES → ERP) 구조의 국제 표준 |
| 19 | OPC-UA | Open Platform Communications Unified Architecture. 산업 자동화 분야의 국제 표준 통신 프로토콜 |
| 20 | DA (Data Access) | OPC-UA 4대 핵심 규격 중 하나. 실시간 센서 값을 읽고 쓰는 기능 |
| 21 | HA (Historical Access) | OPC-UA 4대 핵심 규격 중 하나. 과거 이력 데이터를 기간 조건으로 조회하는 기능 |
| 22 | AC (Alarms & Conditions) | OPC-UA 4대 핵심 규격 중 하나. 조건 충족 시 클라이언트로 이벤트를 비동기적으로 push하는 기능 |
| 23 | Prog (Programs) | OPC-UA 4대 핵심 규격 중 하나. 클라이언트가 서버 측 함수를 원격으로 실행하는 기능 |
| 24 | PubSub 방식 | Publisher-Subscriber 방식. 퍼블리셔가 특정 수신자를 지정하지 않고 데이터를 발행하면 구독자가 수신하는 비연결 지향형 통신 |
| 25 | asyncua | Python 환경에서 OPC-UA 프로토콜을 구현하기 위해 가장 널리 사용되는 오픈소스 라이브러리 |
| 26 | NER (Named Entity Recognition) | 개체명 인식. 텍스트에서 특정 단어·구절을 식별·분류하는 자연어처리 기술 |
| 27 | 바운딩 박스 | 이미지/영상에서 객체를 사각형 박스로 감싸 위치와 크기를 지정하는 어노테이션 방식 |
| 28 | 시맨틱 세그멘테이션 | 이미지 내 모든 픽셀을 사전 정의된 범주로 분류하는 방식 |
| 29 | 인스턴스 세그멘테이션 | 객체를 픽셀 단위로 구분하여 개별 인스턴스 단위로 분류하는 방식 |

> ④ 전송 계층에서의 암호화 및 인증을 지원한다.

> ① ②   &nbsp; ② ③④   &nbsp; ③ ②③   &nbsp; ④ ①④

> [!info]- 💡 정답 및 해설 보기
> **정답**: ③ (②③)
>
> **해설**: ①과 ④는 클라이언트-서버 방식의 특징이다. PubSub은 비연결 지향형으로 연결 상태를 유지하지 않으며(①이 틀림), 암호화·인증 지원은 클라이언트-서버 방식의 언급 특징이다(④가 틀림). PubSub의 핵심 특징은 '퍼블리셔가 수신자를 지정하지 않고 발행(②)'과 'One-to-Many / Many-to-Many 통신 가능(③)'이다.

### 문제 1 `[심화]`

**인더스트리 3.0에서 인더스트리 4.0으로의 전환에서 'Data Silo' 문제가 발생하는 근본 원인을 설명하고, RAMI 4.0과 OPC-UA가 이 문제를 어떻게 해결하는지 그 상호보완적 역할을 논하시오.**

> [!example]- 💡 모범 답안 및 채점 포인트
> **모범 답안**:
>
> 인더스트리 3.0의 Data Silo 문제는 각 설비와 시스템이 특정 물리적 장비에 종속된 독자적 프로토콜과 고립된 수직적 통신 체계를 사용한 데서 비롯된다. 센서, 제어기(PLC), SCADA, MES, ERP 등 각 계층이 서로 다른 언어(프로토콜)로 동작하여, 계층 간 또는 이기종 장비 간 데이터 교환이 극도로 어려웠다.
>
> RAMI 4.0은 이 문제에 대한 '거시적 표준 지도(Map)'로 작동한다. 3축 구조(X: 수명주기, Y: 계층 구조, Z: 아키텍처 계층)를 통해 제조 시스템의 모든 구성 요소를 어디에 위치시키고 어떻게 연결할지에 대한 공통 참조 프레임워크를 제공한다. OPC-UA는 이 프레임워크를 실제로 구현하는 핵심 통신 표준이다. OPC-UA는 Y축의 수직적 통합을 담당하여 현장 센서(Asset 계층)에서 클라우드(Connected World)까지 이기종 장비 간 끊김 없는 안전한 데이터 전달을 보장한다. 나아가 단순 데이터 전송을 넘어 정보 모델링을 통해 데이터에 의미(Context)를 부여함으로써, Z축의 Integration-Communication-Information 계층 간 원활한 데이터 흐름을 지원한다. 결국 RAMI 4.0이 '무엇을 어떻게 연결할 것인가'에 대한 청사진을 제시하고, OPC-UA가 그 청사진을 실제 통신 레벨에서 구현하는 도구로 기능하며 서로 상호보완적으로 Data Silo 문제를 해결한다.
>
> **핵심 채점 포인트**:
> - [ ] Data Silo의 근본 원인: 이기종 프로토콜, 고립된 수직적 통신 체계 명시

### 문제 2 `[심화]`

**제조 AI 데이터셋 구축에서 '데이터 정제' 단계와 '데이터 가공' 단계의 목적과 주요 활동을 비교하고, 두 단계를 구분하지 않고 혼용했을 때 발생할 수 있는 문제점을 실제 제조 현장의 맥락에서 논하시오.**

> [!example]- 💡 모범 답안 및 채점 포인트
> **모범 답안**:
>
> 데이터 정제 단계는 수집 단계의 원시데이터에서 AI 학습에 부적합한 요소를 제거·변환하여 '원천데이터'를 확보하는 활동이다. 주요 활동으로는 형식 변환, 중복 제거, 결측치 처리, 개인정보 비식별화, 이상 데이터 선별 등이 있다. 이 단계의 핵심은 '부적합 제거'이지, 의미 부여가 아니다.
>
> 데이터 가공 단계는 정제된 원천데이터에 AI 학습 목적에 맞는 Ground Truth(정답)와 주석을 부여하여 '학습데이터'로 전환하는 활동이다. 주요 활동으로는 라벨링, 어노테이션(바운딩 박스, 세그멘테이션 등), 구조화 등이 있다. 이 단계의 핵심은 '의미 부여'다.
>
> 두 단계를 혼용하면 심각한 품질 문제가 발생한다. 예를 들어 제조 공정에서 수집된 센서 데이터에 정제(결측치 처리, 이상치 제거) 전에 라벨링을 수행하면, 잘못된 원시 데이터에 '정상(0)' 또는 '이상(1)' 레이블이 부여될 수 있다. 이는 Ground Truth 자체가 오염된 것으로, 이후 학습된 AI 모델이 불량을 정상으로 분류하거나(False Negative) 정상을 불량으로 분류(False Positive)하는 오류를 양산한다. 또한 개인정보 비식별화 전에 가공이 이루어지면 법적·윤리적 리스크가 발생한다. 따라서 수집 → 정제 → 가공의 순서는 데이터 품질과 AI 모델 신뢰성 모두를 위한 필수적인 원칙이다.
>
> **핵심 채점 포인트**:
> - [ ] 정제 단계의 목적(부적합 제거, 원천데이터 확보)과 주요 활동 명시
> - [ ] 가공 단계의 목적(Ground Truth 부여, 학습데이터 전환)과 주요 활동 명시
> - [ ] 두 단계의 핵심 차이점 명확화: '제거' vs '의미 부여'
> - [ ] 혼용 시 발생하는 구체적 문제 (오염된 Ground Truth, False Positive/Negative 등)
> - [ ] 가점: 제조 현장 맥락(공정 센서 데이터, 불량 분류)과 연결한 사례 제시

### 문제 3 `[심화]`

**OPC-UA의 4대 핵심 규격(DA, HA, AC, Prog)이 실제 스마트 팩토리 운영에서 어떻게 상호 연동되어 종합적인 설비 관리를 가능하게 하는지, 구체적인 시나리오를 들어 논하시오.**

> [!example]- 💡 모범 답안 및 채점 포인트
> **모범 답안**:
>
> 스마트 팩토리의 CNC 공작기계 관리를 예로 들어 4대 규격의 상호 연동을 설명한다. 먼저 DA(Data Access)는 현재 주축 회전속도, 절삭 온도, 진동 등의 실시간 값을 1초 주기로 읽어와 운전자의 HMI(Human-Machine Interface)에 표시한다. 평상시 운전 데이터는 HA(Historical Access)로 서버에 이력이 저장되어, 일간/주간 가공 정밀도 추이와 공구 마모 패턴을 분석하는 트렌드 분석에 활용된다. 이 이력 데이터는 예지보전(Predictive Maintenance) 모델의 학습 데이터로도 활용될 수 있다.
>
> 절삭 온도가 기준값(예: 80°C)을 초과하는 순간, AC(Alarms & Conditions) 규격이 즉각적으로 운전자 화면과 관리자 모바일에 알람을 비동기로 push한다. 운전자가 폴링을 기다릴 필요 없이 즉각 인지 가능하다. 만약 알람 조건이 심각 수준(예: 100°C 초과)이거나 예지보전 모델이 즉각 정지를 권고하면, Prog(Programs) 규격을 통해 원격지의 제어 시스템이 해당 기계의 긴급 정지(Emergency Stop) 메서드를 호출한다. 이처럼 DA로 현재 상태를 모니터링하고, HA로 이력을 분석하며, AC로 이상 징후를 즉각 감지하고, Prog로 원격 제어를 실행하는 4대 규격의 연동이 실시간 설비 관리의 완전한 사이클을 구성한다.
>
> **핵심 채점 포인트**:
> - [ ] DA: 실시간 데이터 읽기·쓰기 기능과 활용 맥락 설명
> - [ ] HA: 이력 데이터 조회와 트렌드 분석 활용 설명
> - [ ] AC: 조건 기반 이벤트의 비동기 push 특성과 필요성 설명
> - [ ] Prog: 원격 함수 실행을 통한 능동적 제어 설명
> - [ ] 가점: 4개 규격이 단순 나열이 아닌 하나의 연동 시나리오로 유기적으로 연결된 논리 구조 제시
