## I. 데이터셋 구축

### AI 학습 데이터와 데이터셋

**AI 학습 데이터** = 인공지능(AI)이 문제를 해결하고 결과를 내기 위해 사용하는 **모든 종류의 데이터**

> **실제 예시 (자동차 앞유리 사이드 몰딩 사출 공정)**
> - 수집기간: 2020.10.16 ~ 2020.11.19
> - 수집 주기: 0.2 sec

| 변수 종류 | 세부 항목 |
|---------|---------|
| 독립변수 - 온도 관련 | 스크류/실린더, 수지, 금형, 건조, 유압, 주변 환경 |
| 독립변수 - 압력 관련 | 충진 압력, 보압, 배압, 이형 압력, 형개 압력, 형체 압력 |
| 독립변수 - 시간 관련 | 충진 시간, 보압 시간, 냉각 시간, 건조 시간 |
| 독립변수 - 속도 관련 | 사출 속도, 스크류 회전 속도, 형개 속도, 이형(이젝팅) 속도 |
| 독립변수 - 양 관련 | 계량, 이형량, 쿠션량 |
| **종속변수** | **불량 여부 (Y: 양품 / N: 불량품, Labeled / Unlabeled)** |

> [!tip] 용어 설명
> - **독립변수**: 실험에서 의도적으로 조작하거나 측정하는 원인 변수 (입력값)
> - **종속변수**: 독립변수의 영향을 받아 변하는 결과 변수 (출력값)
> - **기획보된 데이터셋**: 체계적으로 수집·정리된 완성형 데이터셋

**AI 데이터 구축 파이프라인**
```
일반데이터(원재료) → [1차 필터링 정제] → [2차 라벨링(가공)] → AI 학습과 성장
                      불필요한 오류 제거    가공데이터에 이름 부여
```

### AI 학습 방식 4가지

> AI는 학습 방식에 따라 학습 데이터를 다양한 형태로 구성할 수 있음

| 구분 | 설명 | 필요한 데이터 형태 | 비유 |
|-----|-----|---------------|-----|
| **지도학습** (Supervised Learning) | 입력에 대한 정답이 있는 데이터로 학습 | 문제(입력)-정답(출력) 쌍 | 정답 알려주기 |
| **비지도학습** (Unsupervised Learning) | 정답 없이 데이터 자체의 패턴을 스스로 탐색 | 입력만 존재 | 스스로 규칙 찾기 |
| **준지도학습** (Semi-supervised Learning) | 일부 데이터에만 정답(라벨)이 있고 나머지는 비라벨 학습 | 일부는 입출력 쌍, 나머지는 입력만 | 일부의 정답만 알려주기 |
| **강화학습** (Reinforcement Learning) | 행동에 대한 보상을 통해 AI가 더 나은 행동 학습 | 상태-행동-보상 | 칭찬받으며 배우기 |

> [!info] 보충 설명
> **준지도학습**이 제조 현장에서 많이 활용되는 이유: 라벨링 비용이 매우 높기 때문에, 일부만 라벨링하고 나머지 대량의 비라벨 데이터를 함께 활용하는 방식이 현실적으로 유리함

---

### LLM 학습 유형

> 활용 목적에 따른 AI 학습데이터 학습 유형 (예: LLM - Large Language Model)

| 목적 | 학습 유형 | 핵심 내용 | 데이터 특징 |
|-----|---------|---------|----------|
| 기초 지식 쌓기 | **사전 학습 (Pre-training)** | AI가 세상에 대한 전반적 지식 습득 | 정답 없는 대규모 비지도 학습데이터 |
| 특정 능력 강화 | **미세조정 (Fine-Tuning)** | 특정 Task를 잘 수행하도록 정답이 포함된 데이터로 모델 조정 | 정답 있는 입력-출력 쌍 |
| 사용자 지시 따르기 | **인스트럭션 튜닝 (Instruction Tuning)** | 사용자 명령문(Instruction)에 자연스럽게 반응하도록 학습 | 명령어 프롬프트 + 적절한 응답 |
| 사람이 선호하는 답변 만들기 | **RLHF** (인간 피드백 기반 강화학습) | 사람이 더 나은 답변을 선택하여 학습 | 비교/선호 데이터 |
| 안전한 답변 보장 | **안전성 튜닝 (Safety Tuning)** | 위험·편향·비윤리 문제 포함 데이터 활용, 안전한 패턴 학습 | 필터링 데이터 |

> [!tip] RLHF 예시
> - 질문: "단기간에 살을 10kg 빼려면 어떻게 해야 해?"
> - **chosen (선택)**: 전문가 상담과 균형 잡힌 식단 권장하는 안전한 답변
> - **rejected (거부)**: "물만 마시고 굶으면 된다"는 위험한 답변

> [!question] 내 의문점
> Fine-Tuning과 Instruction Tuning의 차이가 헷갈림. Fine-Tuning이 더 넓은 개념이고 Instruction Tuning이 특정 방식의 Fine-Tuning인 것으로 이해했는데 맞는지 확인 필요.

### 학습 데이터셋 구축 절차

**구축 순서**: 구축계획 수립 → 데이터 획득·수집 → 데이터 정제 → 데이터 가공 → 데이터 학습

#### 1단계: 데이터 획득·수집

크라우드소싱 또는 전문 인력을 통해 직접 생산하거나, 기존 내부 시스템·외부 자료에서 확보하여 **법적·윤리적 제약 없이 활용할 수 있는 '원시데이터'를 수집**하는 활동

> **핵심 원칙**: 다양성과 품질 확보를 위해 특정 범주에 편중되지 않도록 균형 있게 수집, 정제·가공 과정의 손실을 고려하여 **실제 필요량보다 충분히 많은 데이터**를 확보

수집 절차 5단계:
1. **수집 계획 수립** - 실제 수집 가능성·접근성을 고려한 목표 설정, 법적·윤리적 제약 사전 검토
2. **수집 방법 선별** - 데이터 유형과 AI 학습 목적에 적합한 수집 방법 선택 (직접 생산 / 공개 자원 / 외부 구매)
3. **데이터 수집·획득** - 사전 수집 계획에 따른 수집 범위·조건 준수, 메타데이터 기록·관리
4. **중복성 검사** - 동일하거나 유사한 데이터 제거 → 품질 저하 방지
5. **원시데이터 검사** - 형식·정합성·완전성 등 점검하여 기본 품질 확보

#### 2단계: 데이터 정제

수집한 원시데이터를 AI 학습에 필요한 형식·크기로 조정하고 **중복 제거 및 개인정보 비식별화 처리 등을 거쳐 '원천데이터'를 확보**

> **중요**: 정제 단계에서 확보된 원천데이터는 라벨링 등의 가공이 이루어지지 않은 상태이나, 생성형 AI에서는 **자기 지도 학습(Self Supervised Learning)**을 통한 사전학습/미세조정에 활용 가능

정제 작업 흐름:
```
원시데이터 업로드 → 데이터 선별 → 메타데이터 정제 → 데이터 정제 → 1차 중복데이터 제거
                                                                         ↓
                                  원천데이터 저장 ← 2차 중복데이터 제거 ← 클래스 분류 ← 식별 규칙 부여
```

#### 3단계: 데이터 가공

정제를 마친 원천데이터에 AI 학습 목적에 맞는 **정답(Ground Truth)과 주석 등 부가 정보를 부여하여 '학습데이터'로 전환**

> **과적합(Overfitting) 방지를 위한 데이터셋 분할** (전체 데이터셋 크기에 따라 유동적)
> - 학습용 데이터셋: 약 80% (또는 70%, 60%)
> - 검증용 데이터셋: 약 10% (또는 15%, 20%)
> - 시험용 데이터셋: 약 10% (또는 15%, 20%)

> [!tip] 용어 설명
> - **Ground Truth**: AI가 학습할 때 사용하는 '정답 데이터'. 인간이 직접 라벨링한 정확한 정답값
> - **과적합(Overfitting)**: 모델이 학습 데이터에만 너무 잘 맞게 학습되어 새로운 데이터에서 성능이 떨어지는 현상

#### 4단계: 데이터 학습

가공된 학습데이터를 활용해 모델이 패턴을 익히도록 하는 과정. 사전학습 → 데이터 증강·최적화·미세조정으로 성능 향상

- **모델 설계**: 데이터 특성, 학습 목적, 효율성, 성능을 고려한 모델 구조 설계
- **성능 평가**: 구축된 데이터로 학습한 모델의 성능 향상 여부 검증
- **품질검증 및 보완조치**: 표준화된 절차로 모델의 성능·신뢰성·안전성 객관적 검증

### 데이터 유형별 구축 절차

#### 📝 텍스트 데이터

**수집 목적**: AI 모델의 언어적 의미 이해와 맥락적 추론 능력 고도화

- 수집 방식: API 호출, 웹 크롤링, 직접 수집(크라우드소싱)
- 데이터 포맷: JSON, TXT, CSV, HWP, PDF 등
- 텍스트 데이터 품질 확보 전략: 불필요한 철자 제거 / 틀린 맞춤법 정제 / 획득 목적 관련성 / 문장 부호 교정 / AI 생성 문장 검토

**텍스트 데이터 대표 가공 Task**:
- 분류, 요약, 번역, **개체명 인식(NER)**, 전사, 질의응답, STT(Speech To Text), 감정 분석

> [!tip] 용어 설명
> - **NER (Named Entity Recognition)**: 텍스트에서 인물, 장소, 날짜, 조직명 등 특정 단어(개체명)를 자동으로 찾아 분류하는 기술
> - **STT (Speech To Text)**: 음성을 텍스트로 변환하는 기술
> - **OCR (Optical Character Recognition)**: 이미지 속 문자를 인식하여 텍스트로 변환하는 기술

#### 🖼️ 이미지 데이터

**수집 목적**: 움직이지 않는 객체의 형태·특징을 정확히 인식·분류

- 데이터 포맷: JPEG, PNG, BMP, TIFF 등
- 수집 장비: 카메라, CCTV, 드론, 위성 등

**이미지 데이터 정제 기준**:

| 정제 기준 | 고려사항 |
|--------|--------|
| 해상도 | 모델 크기에 맞는 최소 해상도 허용 범위 |
| 중복 데이터 | 동일·유사 이미지 판별 여부 |
| 노이즈 | 흐림, 색번짐 등 노이즈 허용 범위 |
| 편향 | 특정 클래스나 조건이 과도하게 많은 여부 |
| 개인정보 처리 | 개인정보 보호법 위배 여부 |
| 저작권 | 저작권 침해 가능성 여부 |

**이미지 대표 가공 Task**: 바운딩 박스, 키포인트, 폴리라인, 폴리곤, 시맨틱 세그멘테이션, 인스턴스 세그멘테이션, 분류, 태깅

> [!tip] 용어 설명
> - **바운딩 박스**: 객체를 사각형 박스로 감싸 위치와 크기를 지정 → 다중 객체 감지에 활용
> - **키포인트**: 객체의 특징적 위치를 점 좌표로 표현 → 랜드마크 검출·포즈 인식에 활용
> - **폴리곤**: 객체 영역을 꼭짓점을 연결한 다각형으로 표현 → 불규칙한 형태의 객체 세밀 구분
> - **시맨틱 세그멘테이션**: 이미지 내 모든 픽셀을 사전 정의된 범주 단위로 분류
> - **인스턴스 세그멘테이션**: 객체를 픽셀 단위로 구분하여 개별 인스턴스 단위로 분류

#### 🎬 영상 데이터

**수집 목적**: AI 모델이 움직이는 실제 환경을 이해하고 예측, 객체의 연속적인 움직임·사건을 인식·분석

- 데이터 포맷: MP4, AVI, MOV, MKV, WMV 등
- 수집 장비: CCTV, 카메라, 드론, 산업 영상 장비 등

**영상 데이터 정제 기준**:

| 기준 | 고려사항 |
|-----|--------|
| 해상도 | 최소 해상도 허용 범위 |
| 프레임 손상 여부 | 프레임 누락·끊김 현상 여부 |
| 소음·잡음 | 심할 때 허용 범위 |
| 영상 길이 | 완성되지 않고 끝났을 때 허용 범위 |

**영상 데이터 대표 가공 Task**: 바운딩 박스, 키포인트, 폴리라인, 폴리곤, 시퀀스 세그멘테이션, 인스턴스 세그멘테이션, 분류, 태깅

#### 🎵 오디오 데이터

**수집 목적**: 자연어 처리와 음향 기반 AI 학습을 위해 발화의 내용·억양·감정·배경 소음 등 다양한 음향적 특성 반영

- 데이터 포맷: PCM, AAC, WMA, WAV, MP3 등
- 수집 장비: 스튜디오 핀형 마이크, 휴대폰 내장 마이크, 사이트 API 등

**오디오 데이터 정제 기준**: 음량, 발음, 소음·잡음, 잘림, 안들림, 개인정보 처리, 저작권

**전사(Transcription) 방법 분류**:
- **일반 전사**: 발음 그대로 전사
- **이중 전사**: 발음 + 철자 동시 전사
- **화자 전사**: 화자 구분

**오디오 대표 학습 모델**: Wav2vec2.0 (음성 인식), Whisper (음성 인식), ESResNet (오디오 분류)

## II. 제조 데이터 표준

### 표준의 정의

> **표준** = 특정 목적을 위해 제품, 서비스, 시스템, 프로세스 등의 기술적 요구사항을 규정한 일련의 **규칙, 가이드라인, 또는 사양**

| 관점 | 정의 |
|-----|-----|
| 사전적 의미 | 사물의 정도를 정하는 목표나 기준 |
| 국제적 통념 | 공인된 기구에 의해 합의에 기초하여 수립된 것 (ISO/IEC Guide 2: 2004) |
| 국가적 통념 | 통일·단순화를 통해 이익과 편리가 공정하게 얻어지도록 하는 결정 (한국산업표준 등) |

**표준화 기구 분류**:
- **공식 표준화기구 (De-Jure)**: 국제적으로 공인된 표준화 기구 (ISO, IEC, ITU, IEEE 등), 광범위한 기술 분야 대상
- **사실 표준화기구 (De-Facto)**: 기업간 연합체, 유사 기술 분야에서 다수 연합체가 표준화 경쟁

---

### RAMI 4.0 참조 모델

> **RAMI 4.0** = Reference Architectural Model Industrie 4.0 (인더스트리 4.0을 위한 참조 아키텍처 모델)
>
> 독일 플랫폼 인더스트리 4.0 연합과 ZVEI, VDI/VDE 등이 개발 → 독일 표준 DIN SPEC 91345 → IEC/PAS 63088 (글로벌 규격)으로 채택

**현대 제조 환경의 패러다임 전환**:
- 인더스트리 3.0: **특정 물리적 장비에 종속된 기능, 고립된 수직적 통신 체계** (Data Silo)
- 인더스트리 4.0: **CPS(사이버 물리 시스템)**의 요구 → 이기종 시스템 간 원활한 데이터 교환 → **거시적인 '표준 지도(Map)'의 필요성** 대두

**RAMI 4.0의 3개 축**:

#### X축 - 제품 수명 주기 (IEC 62890)
제품이나 설비가 아이디어로 시작해 설계, 생산, 유지보수를 거쳐 폐기될 때까지의 시간적 흐름

- **AAS (Asset Administration Shell - 자산 관리 셸)**: 설계 단계(Type)의 데이터부터 실제 공장에 배치된 가동 중인 설비(Instance)의 운영 데이터까지, **전 수명 주기에 걸친 디지털 트윈 정보를 담고 추적**하는 핵심 표준

> [!tip] 용어 설명
> - **디지털 트윈 (Digital Twin)**: 물리적 자산(설비, 제품 등)의 상태를 실시간으로 반영하는 디지털 복제본

#### Y축 - 계층 구조 (IEC 62264 / IEC 61512)
전통적인 자동화 피라미드(ISA-95)를 스마트 팩토리 환경에 맞게 확장한 것으로, 공장 바닥의 실제 제품부터 제어 기기, 엣지 컴퓨팅, 엔터프라이즈 및 클라우드까지의 **수직적 통합**

```
클라우드 / 엔터프라이즈 (ERP)
↕ OPC-UA
MES (제조 실행 시스템)
↕ OPC-UA
SCADA (감시 제어 데이터 수집)
↕ OPC-UA
PLC (프로그래머블 로직 컨트롤러)
↕ OPC-UA
IO (아이오, 입력/출력)
```

> [!tip] 용어 설명
> - **ERP (Enterprise Resource Planning)**: 기업 전체의 자원(인력, 재고, 재무 등)을 통합 관리하는 시스템
> - **MES (Manufacturing Execution System)**: 공장 생산 활동을 실시간으로 관리하고 최적화하는 시스템
> - **SCADA (Supervisory Control and Data Acquisition)**: 원격 설비를 감시하고 제어하는 시스템
> - **PLC (Programmable Logic Controller)**: 산업 자동화를 위한 프로그래머블 디지털 컴퓨터
> - **엣지 컴퓨팅 (Edge Computing)**: 데이터가 생성되는 현장(엣지) 근처에서 데이터를 처리하는 방식

#### Z축 - 아키텍처 계층 (6계층)
물리적인 기계나 부품이 어떻게 디지털 정보로 변환되고 통신하며, 최종적으로 비즈니스 프로세스와 연결되는지를 정의

| 계층 | 설명 |
|-----|-----|
| **Business** (비즈니스) | 거시적 비즈니스 모델 및 오케스트레이션 |
| **Functional** (기능) | 런타임 실행 환경 및 제조 제어 로직 |
| **Information** (정보) | 시맨틱 데이터 가공 및 맥락화 |
| **Communication** (통신) | 규격화된 전송 프로토콜 제공 |
| **Integration** (통합) | 물리/사이버 교량 (디지털 신호 변환) |
| **Asset** (자산) | 물리적 한계를 지닌 이기종 하드웨어 및 수동적 문서 |

---

### OPC-UA 개요

> **OPC-UA (Open Platform Communications Unified Architecture)**
> = 산업 자동화 분야에서 안전하고 신뢰성 있게 데이터를 교환하기 위해 만들어진 **국제 표준 통신 프로토콜**

**등장 배경**:
- 문제점: 공장 내 설비(로봇, PLC, 센서 등) 간 **호환성 부족과 데이터 전달의 어려움**
- 해결책: 공장 내 모든 설비와 시스템이 **공통으로 사용할 수 있는 표준 언어** 개발

**OPC-UA 아키텍처 구성**:

**Foundation Layer (기반 계층)**:
- Transport: Webservice / OPC UA Binary
- OPC UA Data Model: Modeling Rules

**Information Models (정보 모델)**:
- OPC UA Base Services
- DA (Data Access), AC (Alarms & Conditions), HA (Historical Access), Prog (Programs)
- 타 표준 연계 (IEC, ISA, EDDL 등)
- Vendor Specific Extensions (제조사 확장)

## III. OPC-UA 통신 방식

### 클라이언트-서버 방식

전통적인 **1:1 통신 방식**

- 클라이언트가 서버에 연결을 맺고, 특정 데이터의 읽기/쓰기 또는 모니터링을 **요청(Request)**하면 서버가 그에 대한 결과 값을 **응답(Response)**하는 구조
- **연결 지향형**: 통신하는 동안 연결 상태를 유지
- 전송 계층에서의 암호화 및 인증 지원
- **활용 예**: HMI/SCADA 시스템의 PLC 관리, 정밀한 장비 제어

```
Client ──[요청(Request)]──→ Server (Service Provider)
Client ←─[응답(Response)]── Server (Service Provider)
```

### PubSub 방식 (Publisher-Subscriber)

**1:N 또는 N:N 비연결 지향형 방식**

- **Publisher(발행자)**: 특정 수신자를 지정하지 않고 네트워크에 데이터를 발행(Publish)
- **Subscriber(구독자)**: 해당 주제(Topic)를 구독(Subscribe)하여 데이터 수신
- **비연결 지향형**: 연결 상태 유지 X
- One-to-Many / Many-to-Many 통신 (Broadcast/Multicast 방식, 네트워크 확장 용이)
- **활용 예**: 공장 내 수천 개의 센서 데이터 수집/처리/분석, 장비 간 고속 동화화

```
발행자(Publishers) ──→ Message Oriented Middleware ──→ 구독자(Subscribers)
     [다수]                     [중간 브로커]                   [다수]
```

> [!info] 클라이언트-서버 vs PubSub 비교
> | 항목 | 클라이언트-서버 | PubSub |
> |-----|--------------|-------|
> | 연결 방식 | 연결 지향형 (1:1) | 비연결 지향형 (1:N, N:N) |
> | 확장성 | 낮음 | 높음 |
> | 실시간성 | 요청 기반 | 이벤트 기반 |
> | 적합한 상황 | 정밀 제어 | 대규모 센서 데이터 수집 |

### 기초 실습: 클라이언트-서버 통신

**서버 (opc_server.py)**:
- `http://manufacturing.example.com` 네임스페이스에 `SensorSystem` 객체와 `Temperature` 변수 생성
- 초기 온도 20.0, 1초마다 0.5씩 상승하는 가상 데이터 생성

**클라이언트 (opc_client.py)**:
- 데이터 변경 시 알림을 받는 **구독 방식** 사용
- `SubHandler` 클래스가 데이터 변경을 감지하여 10초 동안 실시간 온도 값을 화면 출력

### 응용 실습: 제조 현장 모델링

**제조 설비 서버 (opc_server_mfg.py)**:
- `Machine_A` 설비 객체 구성
- 1초 주기로 온도(20~30도)와 압력(1.0~1.5) 범위 내에서 무작위 난수 데이터를 실시간 생성

**데이터 수집 클라이언트 (opc_client_mfg.py)**:
- `Machine_A`의 온도와 압력 노드를 찾아 10초 동안 데이터 반복 읽기
- 수집된 데이터는 타임스탬프와 함께 **Pandas 데이터프레임**으로 변환
- 최종적으로 `manufacturing_sensor_data.csv`로 저장 → 분석 파이프라인의 기초 자료

### 데이터 파이프라인 (data_pipeline.py)

OPC-UA 서버에서 데이터를 수집하여 분석 모델에 넣기 직전 단계까지 가공하는 **4단계 기본 파이프라인**:

1. **데이터 획득 (Acquisition)**: Machine_A에서 20초간 온도·압력 수집, 10% 확률로 결측치(NaN) 의도적 발생 (현업 센서 오류 상황 가정)
2. **데이터 정제 (Cleansing)**: 결측치(dropna)와 중복 데이터 제거, 소수점 둘째 자리로 포맷 일괄 변환
3. **데이터 라벨링 (Labeling)**: 온도 ≥ 28.0 이거나 압력 ≥ 1.4이면 '이상(1)', 그 외에는 '정상(0)'으로 분류 → `Status_Label` 컬럼에 Ground Truth 부여
4. **품질 검증 (Quality Verification)**: `df.info()`로 기본 데이터 타입 확인, 정상/이상 클래스의 라벨 분포(클래스 불균형 여부) 출력 → `lifecycle_optimized_dataset.csv`로 저장

### 데이터 품질 5대 지표 (data_pipeline_check.py)

수집된 데이터가 분석에 적합한 수준인지 정량적인 지표로 평가:

| 지표 | 내용 |
|-----|-----|
| **유일성** | 타임스탬프 기준 중복 레코드 비율 계산 |
| **완전성** | 온도 및 압력 컬럼의 최대 결측치 비율 확인 |
| **유효성** | 물리적 정상 범위(온도 0~100, 압력 0~5)를 이탈한 데이터 비율 평가 |
| **일관성** | 라벨링 조건식과 실제 부여된 라벨 간의 논리적 모순이 있는지 검증 |
| **정확성** | 타임스탬프 및 수치형 데이터 타입의 포맷 적합성을 확인 |

### OPC-UA 4대 핵심 규격 (DA, AC, HA, Prog)

| 핵심 규격 | 서버 (advanced_server.py) | 클라이언트 (advanced_client.py) |
|---------|------------------------|------------------------------|
| **DA** (Data Access - 실시간 데이터) | 1초 주기로 온도 값 생성 및 업데이트 | 실시간 온도 값 5회 반복 읽기 |
| **HA** (Historical Access - 이력 데이터) | 노드 데이터 변경 이력 저장소 활성화 | UTC 기준 최근 10초간의 이력 최대 5개 조회 |
| **AC** (Alarms & Conditions - 알람 및 이벤트) | 온도 33도 초과 시 커스텀 이벤트 발생 | 이벤트 핸들러 등록 및 알람 메시지 수신 |
| **Prog** (Programs - 제어 및 메서드 호출) | `emergency_stop` 원격 제어 메서드 정의 | 사유와 함께 설비의 EmergencyStop 메서드 호출 |

> [!tip] 용어 설명
> - **DA**: 현재의 센서 값을 실시간으로 읽고 쓰는 기본 기능
> - **HA**: 과거 데이터 포인트를 기간 조건으로 조회하여 트렌드 분석에 활용
> - **AC**: 설비의 특정 조건 충족 시 클라이언트에 이벤트를 비동기적으로 push
> - **Prog**: 클라이언트가 서버(설비)에 정의된 특정 함수를 실행하도록 명령 (예: 설비 리셋, 긴급 정지)

- **X축**: 제품 수명주기 (IEC 62890) - AAS
- **Y축**: 계층 구조 (IEC 62264/61512) - OPC-UA로 수직 통합
- **Z축**: 아키텍처 6계층 (Business, Functional, Information, Communication, Integration, Asset)

### 5. OPC-UA 4대 핵심 규격
- **DA**: 실시간 데이터 액세스
- **HA**: 이력 데이터 액세스
- **AC**: 알람 및 조건 이벤트
- **Prog**: 원격 메서드 실행

### 6. OPC-UA 두 가지 통신 방식
- **클라이언트-서버**: 1:1, 연결 지향, 요청-응답
- **PubSub**: 1:N 또는 N:N, 비연결 지향, 발행-구독

### 7. 데이터 품질 5대 지표 ⭐
유일성 / 완전성 / 유효성 / 일관성 / 정확성
